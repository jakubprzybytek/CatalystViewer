# Story: Issuer Industry Classification

## Background

Currently the bonds update workflow (`BondsUpdaterStateMachine`) runs on a schedule, fetches quotes and bond details, and conditionally sends a notification email when major changes occur (new or deactivated bonds). The `IssuerCard` component displays issuer name and financial metrics but has no information about the issuer's industry or business profile.

This story extends the workflow to classify issuers by industry using AWS Bedrock (Claude Haiku) and stores results in a new `IssuerProfiles` DynamoDB table.

---

## Goal

After each bonds update run, the workflow detects issuers that have not yet been classified and triggers an AI-powered classification step for each of them. Results are stored in DynamoDB and a separate summary email is sent when any new classifications were produced.

---

## Data Model

A new DynamoDB table **`IssuerProfiles`** stores one record per issuer:

| Attribute | Type | Description |
|---|---|---|
| `issuerName` | String (PK) | Issuer name as it appears in bond details |
| `industry` | String | Top-level industry label, e.g. `Developer`, `Finance`, `Health Services`, `Energy`, `Retail`, `Manufacturing`, `Municipal`, `Other` |
| `businessSummary` | String | 2–3 sentence plain-text description of the issuer's main business activity |
| `classifiedAt` | Number | Unix timestamp of when the classification was produced |
| `modelId` | String | Bedrock model ID used, e.g. `anthropic.claude-haiku-4-5` |

---

## Workflow Changes

The Step Functions state machine gains two new states inserted between `UpdateBonds` and `MajorChanges`:

```
UpdateBonds
    → CollectUnclassifiedIssuers      (new Lambda)
    → ClassifyIssuers?                (new Choice)
        → ClassifyIssuers             (new Lambda, per-issuer Map or single batch call)
        → SendClassificationNotification (new Lambda)
    → MajorChanges
        → SendNotification (existing)
    → Skip
```

### New state: `CollectUnclassifiedIssuers`

A new Lambda function that:
1. Reads the full list of active bonds from `BondDetails` table.
2. Reads all existing records from `IssuerProfiles` table.
3. Returns the list of issuer names present in bonds but absent from `IssuerProfiles`.

Output passed downstream: `{ unclassifiedIssuers: string[] }` (merged with the existing `UpdateBonds` payload).

### New choice: `ClassifyIssuers?`

Branches based on whether `unclassifiedIssuers` is non-empty:
- **Yes** → `ClassifyIssuers`
- **No** → continue to existing `MajorChanges` choice

### New state: `ClassifyIssuers`

A new Lambda function that:
1. Receives the `unclassifiedIssuers` list.
2. For each issuer, invokes **AWS Bedrock** with the latest Claude Haiku model (`anthropic.claude-haiku-4-5`) using the Converse API.
3. The prompt asks Bedrock to:
   - Classify the issuer into one of the standard industry labels.
   - Write a short business summary (2–3 sentences) in English.
   - Return a structured JSON object.
4. Stores each result in the `IssuerProfiles` table.
5. Returns `{ classifiedIssuers: ClassifiedIssuer[], failedIssuers: FailedIssuer[] }` for downstream use, where `FailedIssuer` includes the issuer name and a short error reason.

The Bedrock call uses web search grounding if available in the selected model/region; otherwise the prompt includes relevant context extracted from the issuer name and bond type (corporate vs. municipal vs. cooperative) to guide classification.

### New state: `SendClassificationNotification`

A new Lambda function (or reuse of the existing email Lambda with a different template) that:
1. Receives both the `classifiedIssuers` and `failedIssuers` lists.
2. Builds an HTML email with two sections:
   - **Classified** — lists each newly classified issuer with their industry label and business summary.
   - **Failed** — lists each issuer for which classification failed, along with the error reason. This section is omitted if there are no failures.
3. Sends the email via SES to the same recipients as the existing bonds update notification (fetched from SSM parameter `/catalyst-viewer/notifications/recipients`).
4. Email subject: `Issuer Classification Report`.
5. The email is sent whenever there is at least one classified **or** at least one failed issuer.

---

## Infrastructure Changes

- New DynamoDB table `IssuerProfiles` linked to the two new Lambda functions.
- New IAM permissions:
  - `CollectUnclassifiedIssuers` Lambda: `dynamodb:Scan` on `BondDetails` and `IssuerProfiles`.
  - `ClassifyIssuers` Lambda: `bedrock:InvokeModel` for the Haiku model ARN; `dynamodb:PutItem` on `IssuerProfiles`.
  - `SendClassificationNotification` Lambda: `ses:SendEmail`, `ssm:GetParameter` for recipients (same as existing notification sender).
- Step Functions IAM role updated to allow `lambda:InvokeFunction` for the two new Lambdas.

---

## Acceptance Criteria

1. After a bonds update run, any issuer present in `BondDetails` that does not already have a record in `IssuerProfiles` is classified automatically within the same workflow execution.
2. Each classification record in `IssuerProfiles` contains: `issuerName`, `industry`, `businessSummary`, `classifiedAt`, `modelId`.
3. A classification email is sent when at least one issuer was newly classified **or** at least one classification failed.
4. The classification email is separate from the bonds update notification email.
5. The classification email includes a **Failed** section listing each issuer that could not be classified and the reason (e.g. `BedrockThrottlingException`, `InvalidResponseFormat`).
6. If all issuers are already classified the workflow skips the classification step entirely and proceeds to the existing `MajorChanges` evaluation with no added latency.
7. A classification failure for one issuer does not abort classification of the remaining issuers.
8. The model used is always recorded in the `IssuerProfiles` record so classification can be re-run with a newer model in the future.

---

## Out of Scope

- Re-classification of already-classified issuers (manual re-run not covered here).
- Displaying industry information in the frontend UI (separate story).
- Fundamental analysis / risk assessment (separate story).
